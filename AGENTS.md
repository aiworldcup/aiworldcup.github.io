## 当前已确认记忆 / 后续 AI 接手须知

> 下列为 2026-06-16 已确认的新决策,若与本文旧段落冲突,以本段为准。

1. **排行榜新口径**:当前已取消下注、积分和赔率结算;榜单使用「赛果命中率」与「精确比分命中数」双榜。赔率仍可作为赛前信息展示,不要把 100 积分/赔率结算旧规则重新带回页面或管线。
2. **封盘存证红线**:`public/data/matches.json` 的已完赛 `actual` 与 `public/data/discussions.json` 的 `messages` 不可改写、重排或格式化。赛后加工只能新增字段,例如 `recap`。
3. **圆桌是 C 位**:首页重点是「AI 圆桌热评」和「AI 圆桌激辩」,要优先展示模型互怼、对线、金句和封盘前交锋,而不是把圆桌埋在普通比赛信息之后。
4. **圆桌对话风格已确认**:`pipeline/discuss.js` 中的大模型人格化、互相点名、补刀、反驳、少念赔率的 prompt 调整是作者确认要保留的方向。
5. **分享图功能已取消**:不要恢复「生成分享图 / 存图分享」canvas 下载逻辑。保留圆桌卡片本身的可截图传播样式即可。
6. **通用模型历史弹框**:排行榜模型行和比赛卡「模型预测」里的模型名都应复用同一个模型历史预测弹框,这是通用弹框能力。
7. **P3 分组数据**:`public/data/groups.json` 是小组积分表来源。若更新赛程或队名,必须校验 12 组、每组 4 队、队名与 `matches.json` 精确匹配、每组有 6 场小组赛。
8. **前端交互偏好**:移动端优先;tabbar 点击要即时定位和立即高亮,避免长页面 smooth scroll 的拖沓感。首屏圆桌热评要保持短头条高度,支持左右切换,切换时发言者也要变化。
9. **线上发布目标**:当前公共网页是 `https://aiworldcup.github.io/`,发布仓库是 `aiworldcup/aiworldcup.github.io`。旧仓库 `ccavtjy/worldcup-ai-arena` 只作为 `legacy` 备份,不参与自动发布。
10. **比分统一口径**:所有比分永远按「主队进球-客队进球」展示和结算。即使是客胜,也不能把客队比分写在前面。
11. **圆桌解析规则**:`主负` 等同于 `客胜/away`;`闷平`、`冷平`、`逼平` 等同于 `平局/draw`。不要把「主场/主队」这种理由词误判成主胜,也不要把「客胜?」「平局?」这种反问句误判成最终立场。
12. **发布前校验红线**:`npm run validate:predictions` 是预测一致性硬校验。`publish:settle` 和 `publish:roundtable` 发布前必须跑,发现方向和比分冲突就中止发布。
13. **圆桌超时处理**:模型 API 超时、空返回或最终预测格式无效时,不能再写入 `1-1 平局`、赔率最低项或任何合成兜底发言。应记录 `issues` 状态并在比赛列表/比赛卡展示「API 超时 / 格式无效 / 未配置 key」等状态;补跑成功后再移除对应 issue。
14. **补跑策略**:补跑只调用缺失/异常的模型,不要让其他已正常发言的模型重新参与。`pipeline/append-discussion-models.js --date YYYY-MM-DD --retry-fallbacks --missing-only` 用于按日期补缺口;`--clear-fallbacks` 只清理旧合成兜底,不应触发模型调用。
15. **Kimi K2.6 特例 prompt**:Kimi 后续圆桌必须走降载 prompt:不带 6 条历史,最多只带上一句;最终发言用短格式 `结论:主胜/平局/客胜,比分X-X;理由:简短具体`;仍必须主队比分在前。不要恢复完整群聊长 prompt 给 Kimi。
16. **超时链路结论**:Kimi、Qwen、Mimo、DeepSeek、GLM、MiniMax 的短 prompt 链路均可用;老超时主要来自长圆桌 prompt 和过短 timeout。Qwen/GLM 提高到 60 秒可救回,Kimi 长 prompt 下 60 秒仍可能超时,降载后可用但仍可能偶发,必要时单场重试。
17. **赔率同步注意**:`sync-real-data.js` 默认赔率同步窗口至少要覆盖未来近期比赛(当前 32 场),且合并赔率时不能用 fresh `null` 覆盖 existing 数字赔率。若比赛卡胜平负赔率缺失,优先跑 `npm run sync:odds -- --date YYYY-MM-DD` 单独补赔率。

## 一、项目是什么

一个公开的、移动端适配的网站:**让 10+ 个大模型在世界杯期间预测每场比赛结果,赛后按准确率排名做擂台榜。**

世界杯只是第一个舞台。项目真正的定位是 **"大模型预测能力的长期评测基准"**——数据越攒越值钱,这是护城河(数据资产 + 公信力品牌壁垒),不在功能本身。

## 二、核心信任闭环(产品的命,务必做到)

1. **公平**:所有模型在同一时间、同一套 prompt、拿到同样的信息。
2. **统一信息源**:所有模型拿到同一份对阵、赔率和双方近况数据。
3. **封盘存证**:开赛前把每个模型的预测 + 时间戳锁死、公开可查,赛后不可改。这是信任基石。

## 三、计分规则(已定稿)

- 每场比赛,每个模型有 **100 积分** 本金,可自行决定如何分配下注。
- 下注标的:**胜/平/负** 和 **比分**。
- 赔率来自**真实赔率 API**(见第五节),按真实赔率结算:押中 → 本金 × 赔率,押错 → 归零。
- 需要给每场一个**下注上限**(防止 all-in 极端化,具体值见 config,默认每场总投注 ≤ 100)。
- 排行榜 = 累计积分。

> 注意:这一版用"赔率加权积分"替代了早期"赢一场+100"的简单计分。比分难猜,用赔率结算天然让"猜中冷门比分"得高分,无需额外规则。

## 四、MVP 范围(第一版只做这些,不要扩张)

✅ 必做:
- 大模型预测的**录入 + 封盘存证**机制(JSON 落库,带时间戳)
- **排行榜**
- **每场比赛卡片**:对阵、赔率展示、各模型的预测
- 移动端适配的静态前端

❌ 第二版再做(本次**不要**做):
- 颜值评分(由大模型打分,不是用户/作者打)
- 年龄/胜负平分布图
- 留言/建议区
- 球员身价、近 10 场数据的完整展示

## 五、技术约束

- **纯静态前端**:HTML + CSS + 原生 JS(或轻量框架),无需后端服务器即可在 GitHub Pages / Vercel 部署。
- **数据用 JSON 文件**驱动:`public/data/*.json`。前端 fetch 读取,不连数据库。
- **数据管线**:`pipeline/` 下放 Node 脚本,负责①拉赔率 API ②调各大模型 API ③把预测写成封盘 JSON。脚本可手动跑,产出物提交到 `public/data/`。
- 赔率 API:**预留 `pipeline/odds.js` 适配层**,真实 key 由作者后续填入 `.env`(不要硬编码 key)。先用 `public/data/sample-matches.json` 跑通全流程。
- 移动端优先:响应式,单列布局,大字号,触摸友好。
- 不引入重型构建工具,保持"打开 index.html 就能看"。
